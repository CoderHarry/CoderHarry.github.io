React组件基础

1. React虚拟Dom的概念

1.1 什么是虚拟DOM

在React中，render执行的结果得到的并不是真正的dom节点，结果仅仅是轻量级的JavaScript对象，称之为virtual Dom
虚拟DOM是React中的一大亮点，具有batching(批处理)和高效的Diff算法。这让我们可以无需担心性能问题而”毫无顾忌“的随时”刷新“整个页面，由虚拟DOM来确保只对界面变化的部分进行实际的DOM操作。在实际开发中基本上不需要关注虚拟DOM是如何运行的，但是理解其运行机制有助于更好的理解React组件的生命周期，从而进一步优化React的程序。

1.2 虚拟DOM VS 直接操作原生DOM

如何没有virtual Dom，简单说就是直接重置innerHTML。这样在一个大型列表的所有数据变化的情况下比较合理，但是只有一列发生变化，会造成大量浪费。
比较innerHTML和Virtual Dom的重绘过程如下：
innerHTML: render html string + 重新创建所有DOM元素
Virtual DOM: render Virtual DOM + diff + 必要的DOM更新。

和 DOM操作比起来，js计算是非常便宜的。Virtual DOM render + diff显然比渲染html字符串要慢，但是，它依然是纯js层面的计算，比起后面的DOM操作来说，便宜了太多。

React.js相对于直接操作原生DOM有很大的性能优势，很大程度上都要归功于virtual DOM的batching和diff。batching把所有的DOM操作搜集起来，一次性提交给真是的DOM。diff算法时间复杂度也从标准被diff算法的O(n^3)降到了O(n)。

2. CSS模块化

CSS模块化方案很多，主要有2种：

1、 彻底抛弃CSS，使用JS或者JSON来写样式。比如上面讲到的react-style就是属于这一类。

优点：给CSS提供了JS同样强大的模块化处理

缺点：不能使用成熟的CSS预处理器 Sass/Less/Stylus， :hover 和 :active处理非常麻烦。

2、依旧使用CSS，使用JS来管理样式，例如上面所讲到方式来管理，但是这种方式其实会有很多问题，所以需要引入CSS Modules来处理。

2.1 CSS模块化遇到问题

2.1.1 全局污染

CSS使用全局选择器机制来设置样式，优点是方便重写样式，缺点就是全局生效，可能会被覆盖，产生丑陋的!import。CSS中的权重问题，也提高了bug的出现率和使用成本。

2.1.2 命名混乱

由于全局污染的问题，多人开发时会为了避免样式冲突，选择器的命名越来越复杂，非常混乱.

2.1.3 依赖管理不彻底

组件应该相互独立，引入组件的时候，应该只需要引入需要的CSS样式，但是现在做法是引入JS，还需引入CSS，而Sass/less很难对每个组件单独编译出CSS，引入所有的CSS浪费。JS的模块化很成熟了，如果能够让JS来管理CSS是很好的解决办法。Webpack 中 css-loader就提供了这种能力。

2.1.4 无法共享变量

复杂组件要使用JS和CSS来共同处理样式，就会造成变量冗余，Sass/less都不提供跨JS和CSS的共享变量的能力。

2.1.4 代码压缩不彻底

由于移动端网络的不确定，现在对CSS压缩到了一种变态的成都。很多压缩工具为了节省一个字节就会把’16px’转换为’1pc’。但对非常长的class名就无能为力了。

总结

上面遇到的问题CSS自身目前是无法解决的，但是如果通过JS来管理CSS就很好的解决，在React中使用对象样式，其实就是完全抛弃了CSS，在JS中以对象语法来写CSS，这个也是比较蛋疼的~。 直到出现了CSS Modules.

2.2.3 CSS Modules使用技巧

CSS Modules是对现有的CSS做减法。为了追求简单可控，建议遵守以下原则：

  1. 不使用选择器，只是用class名来定义样式
  2. 不层叠多个class， 只是用一个class把所有样式定义好
  3. 所有的样式通过composes组合来实现复用
  4. 不嵌套
